# Security

このドキュメントは、AI駆動開発における秘密情報と本番環境の扱いをまとめます。

## 基本方針

AIエージェントには、秘密情報と本番環境を直接扱わせません。

AIは実装補助、調査、差分修正、レビュー補助として使います。本番DB、本番API、本番環境への反映判断は人間が行います。

## 書いてはいけない情報

リポジトリ、README、docs、Issue、PR、AIへの指示文に、次の実値を書きません。

- `.env` の実値
- APIキー
- DBパスワード
- AWSキー
- アクセストークン
- 秘密鍵
- セッション値
- 本番接続文字列
- 管理画面の認証情報

環境変数名の例を書くことはできますが、値は書きません。

## docsに書いてよい情報

docs に書いてよいのは、次の範囲までです。

- 環境変数名
- 用途
- 取得元
- 必須または任意
- ローカルセットアップ手順
- 値を管理する場所の説明

例:

```text
API_CATALOG_BASE_URL: APIs.guru 取得先URL。値はローカルの .env で管理する。
```

実値は `.env`、パスワードマネージャー、クラウドのシークレット管理など、リポジトリ外で管理します。

## AIに触らせないもの

AIエージェントには次の操作をさせません。

- 本番DBへ接続する
- 本番DBを更新する
- 本番APIを実行する
- 本番環境へSSH接続する
- 本番環境で artisan コマンドを実行する
- 本番環境で migration を実行する
- 本番環境の `.env` を表示または編集する
- 本番環境の秘密情報を読み取る

特に `migrate:fresh`、`db:wipe`、`docker compose down -v`、`docker system prune`、`docker volume prune` のような破壊的操作は、本番環境では実行しません。

## AIへの指示文

AIへの指示文には、秘密情報を含めません。

悪い例:

```text
本番DBの接続情報を使って接続して確認してください。
```

良い例:

```text
DB接続情報の実値は共有しません。ローカル環境で再現できる範囲だけ確認してください。
```

## 秘密情報が混入した場合

秘密情報がリポジトリやPRに混入した場合は、単にファイルから削除するだけでは不十分です。

人間が次の対応を判断します。

- 該当キーやパスワードを無効化する
- 新しい値へローテーションする
- Git履歴の扱いを確認する
- PR、Issue、チャットログなどの露出範囲を確認する
- 必要に応じて関係者へ共有する

AIに秘密情報を貼り付けて調査させないでください。
