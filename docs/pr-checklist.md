# PR Checklist

PR作成前に、次の項目を確認します。

## 目的と差分

- [ ] 目的外の変更がない
- [ ] 不要なリファクタリングが混ざっていない
- [ ] アプリ機能、DB、Docker構成を意図せず変更していない
- [ ] 差分が説明できる

## 責務境界

- [ ] Controller に業務判断やDB操作が混ざっていない
- [ ] Request が形式バリデーションに限定されている
- [ ] Action がユースケース手順に限定されている
- [ ] Service にHTTP都合が混ざっていない
- [ ] Repository に業務判断が混ざっていない
- [ ] Responder に業務判断が混ざっていない
- [ ] Component にデータ取得や業務判断が混ざっていない
- [ ] DTO / Repository / Service / Action の役割が混ざっていない

## DTO / ListDTO

- [ ] DTO境界が崩れていない
- [ ] DTO名が「集約名 + 操作 + DTO」になっている
- [ ] ディレクトリ名に頼ってDTOクラス名から集約名を省略していない
- [ ] DTOにDBアクセスが混ざっていない
- [ ] DTOに業務判断が混ざっていない
- [ ] DTOにHTTPレスポンス生成が混ざっていない
- [ ] DTOにJSON生成が混ざっていない
- [ ] ListDTO が複数DTOの保持と配列化に限定されている
- [ ] snake_case / camelCase の使い分けが境界に合っている

## テスト

- [ ] テストが必要な範囲で追加または更新されている
- [ ] Featureテストでユースケースが固定されている
- [ ] Unitテストで Service / DTO などの境界が必要に応じて確認されている
- [ ] Mock を使う境界が妥当である
- [ ] テストが通っている、または未実行理由を説明できる

## ドキュメント

- [ ] README / docs 更新が必要な変更か確認した
- [ ] 必要な場合は README / docs を更新した
- [ ] MDルーターの追加・削除・修正が必要な変更か確認した
- [ ] Markdownリンクが切れていない
- [ ] Markdownの表示崩れがない

## コメント

- [ ] 通常コメント・PHPDoc・JSDocの説明文が日本語で書かれている
- [ ] コメントが目的、責務、判断理由、制約、処理順、変更時に守る条件を補っている
- [ ] 型やシグネチャから明らかな内容を重複して書いていない
- [ ] 未確認内容、将来予定、推測を書いていない
- [ ] コメントでテスト不足や責務違反を正当化していない

## セキュリティ

- [ ] `.env` の実値が含まれていない
- [ ] APIキーが含まれていない
- [ ] DBパスワードが含まれていない
- [ ] AWSキーが含まれていない
- [ ] 本番DB、本番API、本番環境に直接触れる変更がない
- [ ] AIへの指示文やPR説明に秘密情報が含まれていない

## レビュー準備

- [ ] 何をレビューしてほしいか説明できる
- [ ] 仕様判断が必要な箇所を明示している
- [ ] 既知の未対応事項があれば明示している
- [ ] main 反映前に人間が最終判断する状態になっている
